# AI運用保守マネジメントプラットフォーム

## 02_データ論理設計_Ver1.0

------------------------------------------------------------------------

# 1. 文書概要

## 1.1 目的

本書は、AI運用保守マネジメントプラットフォームにおいて使用する業務データ、マスタデータ、関連情報、AI提案履歴およびシステム管理データの基本構造を定義することを目的とする。

本設計では、以下を重視する。

-   メール受付情報を確実に記録すること
-   Outlook上の元メールおよびメールスレッドとの追跡可能性を確保すること
-   1件の受付に複数の対象システムを関連付け可能とすること
-   1件の受付から複数案件を作成可能とすること
-   複数の受付を同一案件へ関連付け可能とすること
-   案件をシステム単位で独立管理すること
-   人による修正結果を正として保持すること
-   人による修正結果をAIが自動的に上書きしないこと
-   人による修正結果を次回以降のAI提案改善に活用可能とすること
-   過去の対応履歴を長期的な運用保守ナレッジとして活用可能とすること
-   Phase1の受付機能を過度に複雑化せず、将来拡張可能なデータ構造とすること

------------------------------------------------------------------------

# 2. 設計上の情報源および優先順位

本データ設計では、以下の順序で情報を参照する。

1.  最新のアーキテクチャ設計
2.  最新のシステム構成設計
3.  本データ設計ヒアリングにおいて確定した事項
4.  後続設計で新たに確定する事項

上位設計と、その後のデータ設計ヒアリングで確定した内容に矛盾がある場合は、矛盾箇所を明示し、上位設計更新対象として管理する。

要件にない仕様は勝手に補完せず、「要件未定義」として扱う。

ただし、上位設計で確定した要件を実現するために当然必要となる内部ID、日時、状態管理項目等については、データ設計上必要な項目として定義する。

# 3. データ設計原則

## 3.1 人の確定値を優先する

AIによる解析・提案結果をSEが修正した場合、SEによる修正結果を正とする。

AIは、人が修正した確定値を自動的に再変更しない。

ただし、人による修正結果はAIが参照し、次回以降のAI提案改善に活用する。

## 3.2 AI学習・提案改善用フィードバック

本システムにおける「AI学習」は、AIモデル自体を直接再学習させることを前提としない。

過去のAI提案、人による修正、関連解除、再関連付け、案件化判断等の実績データを参照し、次回以降のAI提案品質を改善する方式を基本とする。

SEへの追加負担が発生しない操作結果については、自動的にAI学習・提案改善用データとして蓄積する。

SEへの入力が必要なフィードバックについては、原則として任意回答とし、通常業務を妨げない。

## 3.3 データ重複の抑制

同一情報を複数リストへ重複保存することは原則として避ける。

例えば、AIが提示した類似案件候補は「AI類似提案履歴リスト」で保持し、SEが実際に選択した結果は「受付案件関連リスト」を正として管理する。

AI提案結果とSE判断結果を分析する場合は、両データを突き合わせる。

## 3.4 履歴保持

業務上重要な変更については、現在値のみではなく変更履歴を保持する。

主な対象は以下とする。

-   受付と案件の関連付け
-   受付と案件の関連解除
-   受付と案件の再関連付け
-   案件ステータス変更
-   案件再オープン
-   案件の対象システム変更
-   AI類似案件提案
-   プロンプト変更

## 3.5 Phaseとデータ設計の分離

データ構造として設計することと、Phase1で機能実装することは分けて考える。

将来のAI活用または後続フェーズで必要になるデータ構造を設計上定義する場合があるが、その機能を必ずPhase1で実装することを意味しない。

Phase1の具体的な実装対象は、データ設計完了後の「Phase1機能スコープ整理」で確定する。

------------------------------------------------------------------------

## 3.6 論理データ領域と物理リストの分離

本設計では、業務上の責務を表す「論理データ領域」と、Microsoft Lists上に実装する「物理リスト」を区別する。論理データ領域と物理リストは必ずしも1対1ではなく、保守性、参照方法、データ増加特性および実装負荷を考慮して統合する。

用語は以下に統一する。

- リスト：Microsoft Lists上のデータ格納単位
- レコード：リスト内の1件のデータ
- 列：レコードを構成するデータ項目
- 論理名：設計書・利用者向けの日本語名称
- 物理名：Power Apps、Power Automate、Microsoft Graph、JSON等から参照する英数字PascalCase名称

Microsoft Listsを指して「DB」と呼称しない。

## 4.3 案件化関連の状態管理

「案件化」とは、現行運用における課題管理簿、依頼管理簿、障害管理簿等への新規記載に相当し、本プラットフォームで新規案件を登録することをいう。

案件化判断の結果は「新規案件化」「既存案件への関連付け」「案件化不要」のいずれかとする。「案件化判断完了」と「新規案件化」は同義ではない。

Power Apps上では、人による案件化判断が完了していない状態を「未案件化」と表示する。内部データ上は「未判断」「判断済み」で管理し、表示用語と内部状態を分離する。

# 5. 受付情報リスト

## 5.1 基本方針

1通の受信メールにつき、受付情報リストに1件の受付情報を登録する。

1件の受付には複数の対象システムを関連付け可能とする。

受付件名は元メールの件名を加工せず保持する。

案件化時の案件件名については、案件情報リスト側で管理する。

Outlook上の元メールとの追跡、メール重複判定、関連メール判定および障害復旧時の再処理を可能とするため、必要なメール識別情報および会話関連メタデータを保持する。

## 5.2 主な保持情報

受付情報リストでは、少なくとも以下の情報を管理する。

  ----------------------------------------------------------------------------------------------
  項目                                内容
  ----------------------------------- ----------------------------------------------------------
  受付No                              受付を一意に識別する業務番号

  OutlookメールID                     Outlook上のメールを識別・再取得するための識別情報

  Internet Message-ID                 メール自体の識別、重複判定等に利用する候補情報

  会話識別情報                        Conversation
                                      ID等、同一メールスレッド判定に使用する候補情報

  親受付No                            関連メールを起点メールへ直接紐付けるための受付No

  メール判定種別                      新規受信、リマインドメール、その他関連メール、判定不能等

  件名                                元メールの件名。加工しない

  送信者                              表示名およびメールアドレス

  To                                  表示名およびメールアドレス。複数保持可能

  Cc                                  表示名およびメールアドレス。複数保持可能

  メール送信日時                      メール送信元における送信日時

  メール受信日時                      メールボックスでの受信日時

  受付登録日時                        受付情報リストへの登録日時

  AI最終解析日時                      最後にAI解析を実施した日時

  AI要約                              AIによる受付内容の要約

  その他AI解析情報                    上位設計および後続設計に基づき必要な項目を保持
  ----------------------------------------------------------------------------------------------

具体的なOutlookメタデータの採用項目、重複判定キーおよび関連付け判定キーは、Power
Automate、Microsoft
Graph等で実際に安定取得可能な属性を確認した上で後続設計にて確定する。

## 5.3 メール関連構造

関連メールは、最初の起点メールへ直接紐付ける1階層構造とする。

親→子→孫の多階層構造は採用しない。

起点メール自身の親受付Noは空欄とする。

例：

受付-001 起点メール 親受付No：空欄\
受付-002 関連メール 親受付No：受付-001\
受付-003 リマインドメール 親受付No：受付-001

Power
Appsでは、Outlookのスレッド表示に近い形で、一連の関連メールを確認可能とする。

## 5.4 メール判定

メール判定は、上位設計に基づき以下の順序を基本とする。

1.  メールメタデータによる機械判定
2.  AIによる内容判定
3.  判定不能の場合は人による確認

少なくとも以下の判定種別を想定する。

-   新規受信
-   リマインドメール
-   その他の関連メール
-   判定不能

具体的な判定条件は後続設計で定義する。

## 5.5 メール本文および原本

メール本文およびメール原本の具体的な保存方式については、既存上位設計を正とする。

受付情報リストへの本文全文保存による容量増加を避けることを考慮し、以下の役割を分離する。

-   Outlook上のメール原本
-   受付情報リスト上の業務情報
-   AI解析に必要な情報
-   必要に応じたSharePoint上の保存データ

具体的な保存方式は後続設計で確定する。

## 5.6 AI解析結果

受付情報リストには、業務上必要なAI解析結果およびAI最終解析日時を保持する。

AI解析の成功・失敗に関する詳細な技術情報は、原則としてシステムエラーリストで管理する。

# 6. 受付対象システムリスト

## 6.1 目的

1件の受付に複数の対象システムを関連付け、対象システム単位の案件化判断を管理する。対象システム0件の受付も許可する。

AIによる初回提案値と人による確定値は両方保持し、人の確定値をAIが自動上書きしない。AI初回値は後続の再解析結果で上書きせず、最新AI判定値は原則保持しない。

## 6.2 基本構造

少なくとも以下を管理する。

| 項目 | 内容 |
|---|---|
| 受付・システム関連ID | 関連関係の一意識別子 |
| 受付No | 対象受付 |
| 対象システム | 関連するシステム |
| 判定元区分 | AI初回提案または人の確定等を識別 |
| 案件化判断状態 | 対象システム単位の未判断／判断済み |
| 案件化判断結果 | 新規案件化、既存案件へ関連付け、案件化不要等 |
| 案件化しなかった理由 | 任意入力。後から変更可能 |
| 判断日時 | 人が判断した日時 |
| 判断者 | 判断実行者 |
| 登録日時 | 関連登録日時 |
| 登録者 | AIまたはSE等 |
| 有効状態 | 現在有効な関連かどうか |

具体的な列構成・列型は後続詳細設計で確定する。

## 6.3 追加・無効化・再追加

受付後の対象システム追加を許可する。対象システムを外す場合は物理削除せず関連レコードを無効化する。再追加時は新規関連レコードを作成し、追加・無効化・再追加の経緯を追跡可能とする。

案件化判断済み対象システムを無効化しても、既存の受付案件関連リストは自動解除しない。案件との関連解除は人が別途判断する。

## 6.4 受付全体の案件化判断

人による振り分け完了前は未案件化とする。受付に有効な対象システムがある場合は対象システム単位で案件化判断を行い、必要な振り分けが完了した時点で受付全体を判断済みとする。

対象システム0件の場合は、人が確認して案件化不要と判断した時点で判断済みとする。AIが判定できず0件だった状態と、人が確認して対象システムなしと確定した状態は区別する。

一度判断済みとなった受付は、後から対象システムが追加されても自動的に未判断へ戻さない。新規システムは原則としてプラットフォームへの定義後に受付対象とする。ただし、定義前の過去受付についても必要に応じて後から案件化可能とする。

## 6.5 Power Apps表示

人が未確定の間はAI提案値を参考値であることが分かる形で表示し、同時に未案件化状態であることを明示する。人が確定・修正した後は人の現在確定値を通常画面に表示し、AI初回値は内部データとして保持する。

# 7. 案件情報リスト

## 7.1 基本方針

案件は以下の原則で管理する。

> 1案件 ＝ 1対象システム

1件の受付に複数システムが関連する場合は、必要に応じてシステム単位で別案件を作成する。

そのため、1つの受付Noから複数の案件Noが生成されることを許可する。

## 7.2 案件番号

案件番号は、人が見た時点で案件種別および発生時期を識別できる形式とする。

基本形式は以下とする。

`案件種別-yyyyMMdd-連番`

例：

-   障害-20260709-001
-   依頼-20260709-001
-   QA-20260709-001
-   課題-20260709-001
-   ToDo-20260709-001

具体的な採番ロジックおよび同時実行時の重複防止方式は後続設計で定義する。

## 7.3 主な保持情報

案件情報リストでは、少なくとも以下の情報を管理する。

  項目             内容
  ---------------- --------------------------------
  案件No           案件を一意に識別する業務番号
  案件件名         案件として管理する件名
  案件種別         障害、依頼、QA、課題、ToDo等
  対象システム     1案件につき1システム
  主担当チーム     1チーム
  関連チーム       複数設定可能
  主担当者         1人
  副担当者         0人以上
  案件ステータス   現在の案件状態
  優先度           上位設計に基づく優先度
  SLA関連情報      SLA管理に必要な情報
  AI要約等         上位設計で定義されたAI解析情報
  作成日時         案件作成日時
  更新日時         最終更新日時

その他の具体的な項目は、上位設計に基づきデータ設計および後続設計で定義する。

## 7.4 対象システム変更

関連付けミスまたは調査結果により対象システムが誤っていたことが判明した場合、案件Noを維持したまま対象システムを変更可能とする。

変更前の情報は削除せず、案件変更履歴リストに対象システム変更として保持する。

## 7.5 案件ステータス

案件ステータスはマスタ化する。

保留中の案件は未完了案件として扱う。

具体的な初期ステータスおよび遷移ルールは上位設計を正とする。

## 7.6 完了案件の再オープン

完了案件は再オープン可能とする。

クローズ判断が誤りだった場合、元の案件を再オープンし、案件ステータスを「再オープン」とする。「対応中」には直接戻さない。

一方、過去案件が正しく完了しており、その後に同様の事象が再発した場合は、原則として新規案件として登録する。

これにより、DB拡張対応等の反復的な作業についても、案件発生回数および発生頻度を把握可能とする。

再オープンは案件変更履歴リストに案件ステータス変更として記録する。

# 10. 案件対応履歴リスト

対応内容はPhase1から案件単位の追記型履歴として蓄積する。案件と対応履歴は1対多とする。

少なくとも案件No、対応日時、主対応者、共同対応者、対応内容、AI初回判定対応種別、人の確定対応種別、AI初回判定対応結果、人の確定対応結果、登録日時、登録者、有効状態、関連SharePointリンク等を管理する。

対応日時は原則として書き込んだタイミングの日時を自動登録し、必要に応じて手入力可能とする。対応開始日時・終了日時および下書き状態は設けない。登録者と実際の対応者が異なることを許可する。

主対応者、共同対応者、対応種別、対応結果および対応内容は登録後も変更可能とする。変更履歴は第9章の変更履歴で管理する。

誤登録時は物理削除せず無効化し、Power Apps通常画面では非表示とする。無効化後の再有効化を許可し、その操作履歴を保持する。全面的な誤りの場合は元レコードを無効化し、正しい対応履歴を新規登録可能とする。

# 11. 受付案件関連リスト

## 11.1 目的

受付情報リストと案件情報リストの多対多関係を管理する内部データとする。

以下を可能とする。

-   1受付から複数案件を作成する
-   複数受付を同一案件へ関連付ける

## 11.2 関連ID

受付Noと案件Noを紐付けた際に、各関連関係を一意に識別するための「関連ID」を自動採番する。

関連IDはシステム内部およびAI分析用の識別子とする。

Power Apps上でSEが関連解除等を行う場合、SEに関連IDを直接意識させない。

SEは以下のような業務上自然な操作を行う。

-   受付Noから見て、関連する案件Noを解除する
-   案件Noから見て、関連する受付Noを解除する

内部では対象となる関連IDを特定して処理する。

## 11.3 関連解除

受付と案件の関連は解除可能とする。

解除済みの関連レコードは削除せず保持する。

これにより、過去にどの受付と案件が関連付けられ、その後解除されたかを追跡可能とする。

## 11.4 再関連付け

解除済みの関連を再び有効にする場合、元の関連レコードを再有効化しない。

新しい関連IDを自動採番し、新規関連レコードとして登録する。

これにより以下の履歴を追跡可能とする。

`関連付け → 解除 → 再関連付け`

## 11.5 解除理由

関連解除時には、任意の「AI学習・提案改善用フィードバック」として解除理由を選択可能とする。

解除理由は業務上の必須入力項目ではない。

回答しなくても関連解除を完了できる。

現時点の候補は以下とする。

-   関連なし
-   別案件
-   重複・誤登録
-   その他
-   未回答

回答された場合は、受付内容、AIによる初回紐付け、その後のSEによる修正・再関連付け結果等と組み合わせ、次回以降のAI提案改善に活用する。

## 11.6 関連登録方式

受付と案件を関連付けた際に、AIによる自動関連付けか人による手動関連付けか等を後から識別可能な内部管理項目を保持する。Power Apps通常画面には表示しない。具体的な値は後続設計で確定する。Phase1ではAIは候補提案までとし、最終的な関連付けは人が判断する。

## 11.7 主な保持情報

  項目           内容
  -------------- ----------------------------------------
  関連ID         関連関係の一意識別子
  受付No         対象受付
  案件No         対象案件
  登録者         AIまたはSE等
  関連登録日時   関連付け日時
  状態           有効／解除済
  解除者         解除実行者
  解除日時       解除日時
  解除理由       任意のAI学習・提案改善用フィードバック

# 12. 担当者マスタ

## 12.1 基本方針

担当者マスタは、「人そのもの」の基本情報を管理する論理データ領域である。物理的には共通マスタリスト（`CommonMaster`）内で担当者区分により識別して保持する。

担当者はMicrosoft 365ユーザーを対象とする。

初回登録時にMicrosoft
365から取得可能なユーザー情報を自動取得する方式を第一候補とする。

自動取得機能が現行環境で実現できない場合は、手動で担当者マスタを作成・管理する。

初回登録後はMicrosoft 365ユーザー情報との定期同期を行わない。

## 12.2 主な保持情報

少なくとも以下を保持する。

  項目             内容
  ---------------- -------------------------
  担当者ID         担当者の一意識別子
  表示名           Microsoft 365上の表示名
  メールアドレス   担当者のメールアドレス
  参画日           本運用への参画日
  退場日           本運用からの退場日
  有効状態         有効／無効

その他のMicrosoft 365取得属性については、実機検証後に確定する。

## 12.3 有効状態

担当者の有効状態は以下のルールで自動制御する。

-   参画日前：無効
-   参画日当日：自動有効化
-   参画中：有効
-   退場日当日：有効
-   退場日の翌日以降：自動無効化
-   退場日未設定：参画日以降は有効を維持

無効化後も担当者レコードを削除せず、過去案件および対応履歴との関連を保持する。

## 12.4 再参画

一度退場した担当者の再参画を許可する。

再参画時は既存の担当者マスタを再利用し、参画日、退場日および有効状態を更新する。

過去の対応実績は対応履歴側で保持する。

# 14. AI類似案件提案

## 14.1 基本方針

アーキテクチャ設計でPhase1 AI必須機能として確定しているため、受付情報リストへの登録直後に、AIによる類似案件候補検索を自動実行する。

SEがPower
Apps画面を開いた時点で候補を確認できるよう、事前に解析結果を準備する。

SEによる手動再解析機能は現時点では実装しない。

ただし、データ構造上は複数世代の解析結果を保持可能とする。

## 14.2 検索対象

類似案件候補検索は以下の条件で実施する。

-   受付に紐づく対象システム単位で実行する
-   同一対象システムの案件のみを対象とする
-   保留中を含む未完了案件の全件を対象とする
-   完了済み案件は類似案件候補検索の対象外とする

完了済み案件は類似案件への紐付け候補から除外するが、将来的なナレッジ検索・過去対応参照等では別途活用可能とする。

## 14.3 比較対象

基本的には、新規受付メールの内容と、同一対象システムの未完了案件情報を比較する。

対応履歴や蓄積されたナレッジ全体を毎回類似案件検索の比較対象には含めない。

これはデータ量増加に伴う処理時間、AI利用コストおよび検索負荷の増大を抑制するためである。

具体的な比較項目、AI入力範囲、検索方式および関連度計算方式はAI設計または詳細設計で定義する。

## 14.4 候補数

一定以上の関連性がある案件のみを、最大3件まで候補として提示する。

無理に3件を表示しない。

候補が1件の場合は1件のみ、候補が存在しない場合は「候補なし」とする。

具体的な関連度閾値は現時点では要件未定義とする。

## 14.5 Power Apps上の表示

類似案件候補について、案件件名およびAI要約等を表示する想定とする。

具体的な表示項目、画面レイアウトおよび操作方法等は画面設計で定義する。

## 14.6 AI処理失敗時の再実行

類似案件検索のAI処理に失敗した場合、自動的に1回だけ再実行する。

再実行後も失敗した場合、受付処理自体は継続する。

AI処理失敗によってメール受付全体を停止させない。

# 15. AI類似提案履歴リスト

## 15.1 目的

AIがどの受付に対して、どの対象システムの、どの未完了案件を類似候補として提案したかを保持する。

SEへの追加操作を求めず、AI提案結果と実際のSE判断結果を比較可能とする。

## 15.2 SE判断結果との関係

AIの提案内容は「AI類似提案履歴リスト」に保持する。

SEの実際の判断結果は「受付案件関連リスト」を正とする。

AI類似提案履歴リストにSE選択結果を重複保存せず、必要に応じて両データを突き合わせる。

## 15.3 処理結果

少なくとも以下の3状態を区別する。

  処理結果         意味
  ---------------- --------------------------------------------
  正常・候補あり   AI処理成功。一定以上の関連性を持つ候補あり
  正常・候補なし   AI処理成功。一定以上の関連性を持つ候補なし
  処理エラー       AI処理自体が正常に完了しなかった

「処理エラー」の詳細原因はAI類似提案履歴リストには保持しない。

詳細なエラー原因はシステムエラーリスト側で管理し、開発担当者が確認する。

## 15.4 主な保持情報

  項目                   内容
  ---------------------- --------------------------------------------
  受付No                 対象受付
  対象システム           類似案件検索を実施したシステム
  候補案件No             AIが提示した候補案件
  候補順位               1〜3位
  関連度                 AIが判定した関連性
  関連度閾値             判定時に使用した候補表示閾値
  処理結果               正常・候補あり／正常・候補なし／処理エラー
  AI判定日時             AIによる判定日時
  解析世代               同一受付・対象システムに対する解析世代
  プロンプトバージョン   判定時に使用したプロンプトバージョン

比較対象となった未完了案件数は保持しない。

## 15.5 解析世代

同一受付に対する再解析結果は上書きせず、最大3世代まで保持する。

4世代目が発生した場合は最古の世代を削除し、最新3世代を保持する。

ただし、SE向けの手動再解析機能は現時点では実装対象外とする。

## 15.6 AI学習・提案改善

SEへの追加負担が発生しない以下の結果を自動蓄積し、AI提案改善に利用可能とする。

-   AI候補をSEが選択した
-   AI候補を提示したが、SEがどれも選択せず新規案件化した
-   AIが候補なしと判断し、その後新規案件化した
-   AIが候補なしと判断したが、SEが既存の未完了案件へ紐付けた
-   AI候補とSEの実際の選択案件が異なった

SE判断結果は「受付案件関連リスト」と突き合わせて判定する。

AI類似案件候補に対する「参考になった／参考にならなかった」の個別評価項目は設けない。AI提案候補の採用結果は重複項目を追加せず、受付案件関連リスト等の既存データとの照合により判定する。

# 19. データ保持基本方針

## 19.1 原則保持

現時点では、業務データに一律の保存期限を設けず、原則保持する。

特に以下は長期的な運用保守ナレッジとして重要である。

-   案件情報
-   過去の対応履歴
-   人による最終判断
-   受付と案件の関連情報
-   関連解除・再関連付け履歴
-   案件ステータス変更履歴
-   対象システム変更履歴

## 19.2 単純削除を行わない方針

容量逼迫を理由として、過去の業務ナレッジを単純に削除することは原則として避ける。

データ量が増加した場合は、データの重要度および利用目的に応じて以下を検討する。

-   アーカイブ化
-   通常運用領域からの分離
-   AI分析用補助データの削減
-   不要な中間データの整理
-   不要列・不要データの整理
-   保存先の変更

具体的な対象データおよび保持期間は現時点では要件未定義とする。

# 21. Phase1スコープに関する方針

本データ設計では、将来のAI活用・提案改善を考慮したデータ構造を設計する。

ただし、データ設計上存在する機能・データをすべてPhase1で実装することは前提としない。

Phase1の主目的は受付機能を中心とする基盤構築である。

AI提案機能や利便性向上機能については、開発期間への影響を考慮し、データ設計完了後にPhase1実装対象を整理する。

------------------------------------------------------------------------

# 22. 後続タスク

## 22.1 Phase1機能スコープ整理

現在までに設計した機能を以下に分類する。

-   Phase1の受付機能に必須
-   Phase1で実装するAI機能
-   Phase1では実装せず後続フェーズへ延期
-   将来検討

特にAI提案系機能については、Phase1の開発期間を不必要に延ばさないことを重視する。

## 22.2 データ保持・アーカイブ方針整理

データを以下の観点で分類する。

-   長期保持すべき業務ナレッジ
-   アーカイブ可能なデータ
-   一定期間後に削減可能なAI分析用補助データ
-   容量逼迫時に整理可能な中間データ
-   通常運用領域から分離可能なデータ

案件情報や過去の対応履歴等の業務ナレッジは原則保持する。

古いAI中間解析情報やAI提案履歴等については、実際の利用価値、AI改善効果および容量影響を確認した上で判断する。

## 22.3 上位設計更新

データ設計完了後、上位設計との整合性を再確認する。

上位設計に修正が必要な場合は、アーキテクチャ設計およびシステム構成設計の更新版を作成する。

## 22.4 データ設計レビュー

以下を重点的に確認する。

-   上位設計との整合性
-   本ヒアリングで決定した内容の反映漏れ
-   未確定事項を確定事項として記載していないか
-   不要なデータ重複
-   Microsoft Listsでの実装可能性
-   Power Appsでの操作性
-   Power Automateでの処理負荷
-   将来のAI分析への利用可能性
-   データ量増加時の性能および容量影響

# 23. 現時点の要件未定義事項

以下は現時点では確定しない。

-   Outlookメタデータの最終採用項目
-   メール重複判定キー
-   メール関連付けの具体的な判定キー・ロジック
-   メール本文および原本の具体的な保存方式
-   AI類似案件候補の具体的な関連度閾値
-   AI類似性判定の具体的なアルゴリズム
-   AIへの具体的な入力項目および入力データ量
-   Power Apps上の類似案件候補表示レイアウト
-   各Microsoft Listsの具体的な列型
-   インデックス設計
-   SharePoint Onlineの具体的なアーカイブ方式
-   AI分析用補助データの具体的な保存期間
-   受付案件関連リストの関連登録方式の具体的な値
-   案件変更履歴リストの具体的なカラム構成
-   AI初回判定値と人の確定値の物理的な保持方式
-   対応履歴と複数SharePointリンクの具体的なデータ構造
-   将来履歴を削除する場合の具体的な保持期間・削除条件
-   容量逼迫を判断する具体的な閾値

これらは、上位設計、データ設計レビュー、AI設計、画面設計または詳細設計で決定する。なお、Phase1のAI必須機能範囲は上位設計を正とし、本章の要件未定義事項には含めない。
