# 1. 文書概要

## 1.1 目的

本書は、AI運用保守プラットフォームにおけるテストデータ設計を定義することを目的とする。

結合テスト、受入テストおよび将来実施する各種試験で利用するテストデータの作成方針、データ構成、マスタデータ、業務データ、AI試験データ、異常系データ、管理方法および運用ルールを定義し、追加説明なしで試験担当者が再現性のある試験を実施できる品質を目標とする。

本設計書は「08_テスト基本設計_Ver1.0」「09_結合テスト仕様書_Ver1.0」および「10_受入テスト仕様書_Ver1.0」を補完する設計書として位置付ける。

-   テストデータは再現性を最優先とする。
-   正常系・異常系・境界値を考慮する。
-   Microsoft Listsの実データ構造に合わせたデータを作成する。
-   Outlookメール・SharePoint添付ファイル・Teams通知まで含めた試験を可能とする。
-   Power Apps・Power Automate・AI Builder・Microsoft
    Lists間の連携試験を実施可能とする。
-   AI試験では要約、ToDo生成、判定、AI信頼度、AI再解析を確認できるデータを準備する。
-   CorrelationIdおよびFlowRunIdにより試験結果を追跡可能とする。
-   約20名規模の社内利用を想定した現実的なデータ量とする。
-   Microsoft LearnおよびPower Platformベストプラクティスを優先する。

# 2. テストデータ方針

## 2.1 目的

本章では、AI運用保守プラットフォームにおいて利用するテストデータの作成方針および管理方針を定義する。

結合テスト、受入テストおよび将来実施する回帰試験において、再現性・追跡性・保守性を確保できるテストデータを準備することを目的とする。

------------------------------------------------------------------------

## 2.2 基本方針

本システムでは以下の方針でテストデータを作成する。

-   実運用に近いデータ構成とする。
-   テスト実施者が何度でも再現できるデータとする。
-   正常系・異常系・境界値を網羅する。
-   Microsoft Listsの物理設計に準拠する。
-   CorrelationIdおよびFlowRunIdを利用し、試験結果を追跡可能とする。
-   AI試験専用データを準備する。
-   Outlook、Power Apps、Power Automate、AI Builder、SharePoint
    Online、Microsoft Teamsまで含めた一連の試験が可能な構成とする。
-   Microsoft LearnおよびPower Platformベストプラクティスを優先する。

------------------------------------------------------------------------

## 2.3 テストデータ分類

  分類                 内容
  -------------------- --------------------------------------------------
  マスタデータ         対象システム、案件種別、ステータス等の共通データ
  業務データ           受付情報、案件情報、担当者情報等
  AI試験データ         AI解析用メール、添付ファイル、期待結果
  異常系データ         入力不備、AI失敗、通信異常等
  通知データ           Teams通知、SLA通知確認用
  添付ファイルデータ   SharePoint保存確認用
  ログ・監査データ     CorrelationId、FlowRunId確認用

------------------------------------------------------------------------

## 2.4 テスト観点

  観点       内容
  ---------- ----------------------------
  正常系     通常業務が成立するデータ
  異常系     エラー処理を確認するデータ
  境界値     最大・最小・桁数上限等
  AI         AI解析・信頼度・再解析
  通知       Teams通知・メール通知
  SLA        期限計算・期限超過
  権限制御   利用者権限別データ
  監査       ログ・追跡情報

------------------------------------------------------------------------

## 2.5 テストデータ作成方針

-   受付データは実際の問い合わせメールを想定した内容とする。
-   案件データは受付情報との関連付けを考慮する。
-   添付ファイルは業務で利用する一般的なファイル形式を採用する。
-   AI試験では判定しやすいデータだけでなく、判定が困難なデータも含める。
-   AI再解析試験が可能となるよう、解析済みデータも準備する。
-   CorrelationIdおよびFlowRunIdを付与し、全試験データを追跡可能とする。

------------------------------------------------------------------------

## 2.6 AI試験データ方針

  試験項目           確認内容
  ------------------ -------------------------------
  AI要約             約100文字の要約生成
  ToDo生成           対応内容提案
  受付種別判定       問い合わせ・障害・依頼等
  対象システム判定   対象システム候補抽出
  優先度判定         緊急度・重要度
  AI信頼度           信頼度保存・表示
  AI再解析           再解析結果更新
  AI失敗             業務継続およびSystemError登録

------------------------------------------------------------------------

## 2.7 テストデータ品質基準

-   再現性があること。
-   試験目的が明確であること。
-   重複や不要データがないこと。
-   実運用に近い内容であること。
-   AI判定結果の確認が可能であること。
-   個人情報を含まないこと。

------------------------------------------------------------------------

## 2.8 個人情報・機密情報

本試験では実データを使用しない。

以下を使用する。

-   ダミー利用者
-   ダミーメールアドレス
-   ダミー電話番号
-   ダミー会社名
-   ダミー添付ファイル

個人情報および機密情報はテストデータへ含めない。

------------------------------------------------------------------------

## 2.9 テストデータライフサイクル

``` text
設計
  │
  ▼
作成
  │
  ▼
投入
  │
  ▼
試験実施
  │
  ▼
結果確認
  │
  ▼
必要に応じて初期化・再投入
```

各試験で同一条件を再現できるよう、投入データは管理台帳で管理する。

------------------------------------------------------------------------

## 2.10 設計者判断

本システムは約20名規模の社内利用を前提とするため、大量データによる負荷試験用データではなく、実運用を想定した品質確認用データを中心に整備する。

また、すべての試験データについて CorrelationId および FlowRunId
を利用し、Power Apps、Power Automate、Microsoft Lists、AI
BuilderおよびSystemErrorまで一連の処理を追跡可能とする。
