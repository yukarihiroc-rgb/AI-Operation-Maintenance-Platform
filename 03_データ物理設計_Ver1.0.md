# 1. 文書概要

## 1.1 目的

本書は、AI運用保守プラットフォームで利用する Microsoft Lists
の物理設計を定義する。

データ論理設計で定義した業務データ構造を、Microsoft Lists、Power
Apps、Power Automate および AI
機能として実装可能なレベルまで具体化し、実装・保守時の標準設計書とする。

また、本章は各リスト固有の設計に共通する実装ルールを定義する **Microsoft
Lists 実装標準書**
と位置付け、第2章以降では本章を参照することを原則とする。

## 1.2 適用範囲

本章は以下を対象とする。

-   文書共通方針
-   命名規則
-   共通列
-   Microsoft Lists 設計方針
-   Lookup 設計方針

各リスト固有の列定義、インデックス設定および画面仕様は第2章以降で定義する。

## 1.3 情報源

本書は以下の優先順位で設計する。

1.  要件定義
2.  アーキテクチャ設計 Ver1.12
3.  システム構成設計 Ver1.12
4.  データ論理設計 Ver1.0
5.  本データ物理設計

上位設計と矛盾する内容を発見した場合は推測で補完せず、不整合内容・影響範囲・修正提案を整理した上で上位設計の修正を提案する。

## 1.4 設計判断ポリシー

-   Microsoft Learn および Microsoft 365 / Power Platform
    の推奨事項を優先する。
-   Microsoft Lists
    は業務データストアとして利用し、通常運用では直接編集しない。
-   業務データの更新は Power Apps および Power Automate のみが行う。
-   AI
    の判定結果は参考情報として保持し、人による確定値を正式データとする。
-   業務データは物理削除せず、無効化管理を基本とする。
-   要件へ影響しない実装詳細は保守性・可読性・拡張性を考慮して設計者判断で決定する。

# 2. 共通設計

## 2.1 命名規則

### リスト名

-   PascalCase を採用する。
-   単数形で命名する。
-   業務責務を表す名称とする。

例

  論理名             物理名
  ------------------ --------------
  受付情報リスト     IntakeInfo
  案件情報リスト     CaseInfo
  共通マスタリスト   CommonMaster

### 列名

-   PascalCase を採用する。
-   英数字のみ使用する。
-   接尾辞を統一する。

  接尾辞   用途
  -------- --------------
  Id       GUID・関連ID
  At       日時
  Is       真偽値
  No       業務番号
  Name     名称

### GUID方針

-   システム内部識別子は GUID を採用する。
-   GUID は変更不可とする。
-   業務番号とは分離して管理する。

## 2.2 共通列

原則として全リストに以下の監査列を設ける。

  列名        内容       更新主体
  ----------- ---------- -----------------------------
  RecordId    GUID       システム
  CreatedAt   登録日時   システム
  CreatedBy   登録者     システム
  UpdatedAt   更新日時   システム
  UpdatedBy   更新者     システム
  IsActive    有効状態   Power Apps / Power Automate
  Remarks     備考       必要時

監査列は業務列と分離し、業務ロジックで直接更新しない。

## 2.3 Microsoft Lists設計方針

-   Microsoft Lists は本システムの業務データストアとして利用する。
-   通常利用者による直接編集は禁止する。
-   業務更新は Power Apps を経由する。
-   自動更新は Power Automate を経由する。
-   1リスト1責務を原則とする。
-   業務上必要な履歴は物理削除せず保持する。
-   Choice 列は固定値のみとし、変更可能な値は共通マスタで管理する。

## 2.4 Lookup設計方針

-   リレーションは Lookup 列を基本とする。
-   関連先は内部ID（GUID）で管理する。
-   多段 Lookup は採用しない。
-   循環参照は禁止する。
-   一覧表示に必要な項目は Power Apps 側で取得・表示する。
-   パフォーマンスを考慮し、Lookup の乱用を避ける。

## 2.5 Person列設計方針

### 基本方針

-   Person列は Microsoft 365 ユーザーを参照する場合のみ利用する。
-   業務上の担当者情報として利用し、権限制御には使用しない。
-   主担当者は単一選択、共同担当者は複数選択を基本とする。
-   退場済み担当者も履歴参照可能とするため、必要に応じて担当者関連リストと組み合わせて利用する。

### 利用基準

  用途             Person列
  ---------------- ---------------------
  主担当者         ○
  副担当者         ○
  作成者・更新者   システム監査列
  チーム           ×（共通マスタ参照）

## 2.6 添付ファイル設計方針

### 基本方針

-   Microsoft Lists の添付ファイル機能は原則使用しない。
-   添付ファイルは SharePoint Online ドキュメントライブラリへ保存する。
-   Lists には保存先URLまたは関連情報のみ保持する。

### 理由

-   バージョン管理を容易にする。
-   容量管理を一元化する。
-   Power Apps・Power Automateとの連携を容易にする。

## 2.7 AI共通設計

AIを利用するリストでは、必要に応じて以下の列を採用する。

  列名             内容
  ---------------- ----------------
  AIConfidence     AI信頼度
  AIResultJson     AI応答JSON
  AIModel          使用モデル
  PromptVersion    プロンプト版数
  AIExecutedAt     AI実行日時
  ProcessingTime   AI処理時間(ms)

AI判定値は参考情報であり、人による確定値を上書きしてはならない。

## 2.8 エラー管理共通設計

自動処理では以下を標準採用する。

  列名            内容
  --------------- ------------------
  CorrelationId   一連の処理識別子
  ErrorGroupId    関連エラー識別子
  ExceptionJson   詳細例外情報

業務画面には表示せず、保守・障害解析用途で利用する。

## 2.9 共通運用ルール

-   Microsoft ListsはDBとして利用する。
-   通常利用者はListsを直接編集しない。
-   Power Appsを唯一の業務更新画面とする。
-   Power Automateを唯一の自動更新手段とする。
-   AIは補助機能であり、人の判断を優先する。
-   データは物理削除せず、無効化管理を基本とする。

## 2.10 共通パフォーマンス設計

### 基本方針

-   Delegationを考慮したPower Apps設計を行う。
-   インデックス対象列で検索・並び替えを実施する。
-   必要列のみ取得する。
-   Lookup列の過剰利用を避ける。
-   長文列は一覧表示・検索条件に使用しない。

### 設計基準

-   主要検索列はインデックス対象とする。
-   フィルター条件は委任可能な関数を利用する。
-   AI用JSON列は一覧取得対象外とする。

## 2.11 Microsoft Lists制約事項

### 基本方針

Microsoft Lists の仕様および SharePoint Online
の制約を考慮し、設計時点で性能・保守性を確保する。

### 設計ルール

-   一覧表示しきい値を考慮し、検索・並び替えに利用する列はインデックスを設定する。
-   Choice
    値は固定値のみとし、運用で変更される値は共通マスタで管理する。
-   複数行テキスト列は検索キーとして使用しない。
-   添付ファイル機能は使用せず、SharePoint
    ドキュメントライブラリを利用する。
-   リスト間の多段 Lookup は採用しない。
-   業務データの物理削除は行わず、IsActive による無効化を基本とする。

## 2.12 Power Apps実装ルール

### 基本方針

Power Apps は唯一の業務更新画面とする。

### 実装標準

-   直接 Lists を更新しない。
-   データ更新はフォームまたは Patch を用いて実施する。
-   Delegation 対応関数を優先して使用する。
-   検索・絞り込みはインデックス列を利用する。
-   エラーメッセージは利用者向け表示と詳細ログを分離する。
-   画面では論理名を表示し、物理名は実装内部のみで利用する。

## 2.13 Power Automate実装ルール

### 基本方針

Power Automate は自動登録・自動更新・AI連携を担当する。

### 実装標準

-   フローは冪等性を考慮し、再実行時の重複登録を防止する。
-   GUID をキーとして処理対象を識別する。
-   例外発生時は SystemError リストへ記録する。
-   CorrelationId を引き継ぎ、一連の処理を追跡可能とする。
-   タイムアウト・一時エラーは必要に応じて再試行を利用する。
-   業務更新と通知は可能な限り分離する。

## 2.14 AI実装ルール

### 基本方針

AI は業務判断を補助する機能とし、人による確定値を正とする。

### 実装標準

-   AI判定値と人の確定値を分離して保持する。
-   AIConfidence と AIResultJson をセットで保持する。
-   使用した AI モデルおよび PromptVersion を記録する。
-   AI 処理失敗時は受付処理全体を停止させない。
-   AI の詳細例外は SystemError リストで管理する。
-   AI による更新は人の確定値を上書きしない。

# 3. 受付系リスト

受付系リストは、問い合わせ受付から案件化判断までの情報を管理する。

本章では以下の3リストを定義する。

  論理名                   物理名               用途
  ------------------------ -------------------- ------------------------------
  受付情報リスト           IntakeInfo           受付情報の管理
  受付対象システムリスト   IntakeTargetSystem   受付と対象システムの関連管理
  受付案件関連リスト       IntakeCaseRelation   受付と案件の関連管理

------------------------------------------------------------------------

# 3.1 受付情報リスト（IntakeInfo）

## 3.1.1 目的

問い合わせ受付の起点となる情報を管理する。Outlookメール、Power
Appsからの手動受付等、受付経路を問わず1受付につき1レコードを保持する。受付情報は案件とは分離し、案件化前の状態を保持する。

## 3.1.2 主な利用機能

-   Outlookメール受付
-   Power Apps手動受付
-   AI受付解析
-   AI要約
-   AI ToDo生成
-   類似案件検索
-   案件化判断
-   SLA管理
-   Teams通知

## 3.1.3 リレーション

  IntakeCaseRelation    1:N    案件関連
