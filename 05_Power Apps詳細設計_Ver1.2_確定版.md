# 02_Power Apps共通設計_Ver1.1

# 2. Power Apps共通設計

## 2.1 目的

本章では、AI運用保守プラットフォームで利用する Power Apps
アプリ全体の共通実装方針を定義する。

画面ごとの差異を最小化し、保守性・可読性・性能を確保するため、Power
Fx、状態管理、コンポーネント、データ取得、エラー処理の標準を定義する。

------------------------------------------------------------------------

# 2.2 基本方針

Power Apps は利用者インターフェースおよび画面制御を担当する。

業務処理、データ整合性保証、AI処理、通知処理は Power Automate
が担当する。

## 採用方針

-   Canvas App を採用する。
-   Microsoft Lists をデータソースとして利用する。
-   業務更新処理は Power Automate 経由を原則とする。
-   共通処理はコンポーネント化する。
-   Power Fx は可読性を優先する。
-   画面単位の独自実装を避ける。
-   Delegation 不可となる実装を避ける。

------------------------------------------------------------------------

# 2.3 アプリ構成標準

## App構成

  領域         内容
  ------------ --------------------
  App          共通初期化
  Screen       画面表示・画面制御
  Component    共通UI
  Flow         業務処理
  Collection   一時データ

------------------------------------------------------------------------

# 2.4 App.OnStart標準

App.OnStartでは以下を実施する。

-   利用者情報取得
-   権限情報取得
-   共通変数初期化
-   Theme読込
-   初期Collection生成

例:

``` powerfx
Set(gblCurrentUser, User());
Set(gblIsLoading, false);
Set(gblCorrelationId, GUID());
```

業務データ取得はOnStartで大量取得しない。

理由:

-   初期表示速度低下防止
-   不要データ取得防止
-   Delegation問題回避

------------------------------------------------------------------------

# 2.5 グローバル変数命名規則

## Prefix

  種類              Prefix   例
  ----------------- -------- -----------------
  Global Variable   gbl      gblCurrentUser
  Local Variable    loc      locSelectedItem
  Collection        col      colMenu
  Component         cmp      cmpToast

------------------------------------------------------------------------

# 2.6 グローバル変数一覧

  変数名             型        用途
  ------------------ --------- ------------------
  gblCurrentUser     Record    ログインユーザー
  gblCorrelationId   GUID      処理追跡ID
  gblIsLoading       Boolean   ローディング状態
  gblUserRole        String    権限
  gblTheme           Record    テーマ情報
  gblErrorMessage    String    共通エラー

------------------------------------------------------------------------

# 2.7 画面変数標準

画面固有値はloc変数を利用する。

例:

  変数                用途
  ------------------- --------------
  locSelectedRecord   選択レコード
  locMode             表示モード
  locSearchText       検索条件
  locPageNo           ページ番号

------------------------------------------------------------------------

# 2.8 Collection利用方針

Collectionは以下用途のみ利用する。

-   一時表示データ
-   キャッシュデータ
-   選択状態管理
-   マスタキャッシュ

業務データの永続保存には利用しない。

------------------------------------------------------------------------

# 2.9 Power Fxコーディング規約

## 基本ルール

-   1処理1目的とする。
-   複雑なFormulaは分割する。
-   コメントを記載する。
-   変数名は意味を持たせる。

## 推奨関数

-   With()
-   Concurrent()
-   IfError()
-   Notify()
-   Patch()
-   Refresh()

------------------------------------------------------------------------

# 2.10 Concurrent利用方針

独立したデータ取得はConcurrentを利用する。

例:

``` powerfx
Concurrent(
 Refresh(IntakeInfo),
 Refresh(CaseInfo)
)
```

以下の場合は禁止。

-   前処理結果に依存する処理
-   更新順序が必要な処理

------------------------------------------------------------------------

# 2.11 Delegation対応方針

Microsoft Lists利用時はDelegation警告を発生させない。

推奨:

-   Filter()
-   LookUp()
-   SortByColumns()

大量データ取得時はサーバー側検索を優先する。

------------------------------------------------------------------------

# 2.12 Refreshタイミング

  契機         処理
  ------------ --------------------
  画面表示     必要データ取得
  更新ボタン   Refresh実行
  保存成功     対象データRefresh
  画面離脱     不要キャッシュ破棄

------------------------------------------------------------------------

# 2.13 SaveData / LoadData設計

利用対象:

-   表示件数
-   ソート条件
-   検索条件
-   個人表示設定

業務データ保存には利用しない。

例:

``` powerfx
SaveData(
 colUserSettings,
 "UserSettings"
)
```

------------------------------------------------------------------------

# 2.14 Power Automate Run標準

Input:

  項目            内容
  --------------- ------------

  UserEmail       利用者
  RecordId        対象ID
  RequestJson     要求内容

Output:

  項目      内容
  --------- ------------------
  Success   成功可否
  Message   利用者メッセージ
  ErrorId   エラー識別子

------------------------------------------------------------------------

# 2.15 CorrelationId管理

CorrelationIdは以下へ引き継ぐ。

-   Power Apps
-   Power Automate
-   SystemError

処理開始時に生成し、処理終了まで変更しない。

# 2.16 共通UI利用方針

以下を共通Component化する。

-   Header

-   権限なしメニューは非表示

-   AI内部JSONは一般利用者へ表示しない

# 03_データアクセス設計_Ver1.1

# 3. データアクセス設計

## 3.1 目的

本章では、Power AppsからMicrosoft ListsおよびPower
Automateへアクセスする方式を定義する。

データ取得・更新方式を統一し、性能、保守性、セキュリティを確保することを目的とする。

------------------------------------------------------------------------

# 3.2 基本方針

Power Appsは画面表示および利用者操作を担当し、業務データ更新はPower
Automateを経由する。

## 基本ルール

-   Microsoft Listsを直接編集しない。
-   参照処理はPower Appsから実施可能とする。
-   更新処理はPower Automate経由を原則とする。
-   AI処理結果は参照専用とする。
-   大量データ取得を避ける。
-   Delegation対応を優先する。

------------------------------------------------------------------------

# 3.3 データアクセス方式

  処理       方式
  ---------- -----------------------------------------------
  一覧表示   Power Apps → Microsoft Lists
  詳細表示   Power Apps → Microsoft Lists
  業務更新   Power Apps → Power Automate → Microsoft Lists
  AI処理     Power Apps → Power Automate
  通知       Power Automate

------------------------------------------------------------------------

# 3.4 データソース一覧

  データ               用途               アクセス
  -------------------- ------------------ ------------
  IntakeInfo           受付情報           参照
  IntakeTargetSystem   対象システム判断   参照・更新
  CaseInfo             案件情報           参照
  CaseAssignee         担当情報           参照
  IntakeCaseRelation   関連付け           更新
  SystemError          エラー情報         参照

------------------------------------------------------------------------

# 3.5 参照処理標準

## Filter

検索条件はサーバー側Filterを優先する。

例:

``` powerfx
Filter(
    IntakeInfo,
    Status = locStatus
)
```

------------------------------------------------------------------------

## Sort

SortByColumnsを利用する。

例:

``` powerfx
SortByColumns(
    colResult,
    "Created",
    SortOrder.Descending
)
```

------------------------------------------------------------------------

# 3.6 更新処理標準

業務データ更新はPower Automateで実施する。

Power Apps:

-   入力チェック
-   実行要求
-   結果表示

Power Automate:

-   更新処理
-   排他チェック
-   エラー処理
-   ログ登録

------------------------------------------------------------------------

# 3.7 Patch利用方針

Power Apps内部状態更新以外ではPatchを限定利用する。

利用可能:

-   一時Collection
-   画面状態

原則禁止:

-   業務リスト更新
-   案件作成
-   受付更新

------------------------------------------------------------------------

# 3.8 Lookup取得方針

Lookup値はID保持を基本とする。

表示時:

Masterテーブルから名称取得する。

例:

-   SystemId
-   StatusId
-   UserId

------------------------------------------------------------------------

# 3.9 データキャッシュ方針

キャッシュ対象:

-   共通マスタ
-   メニュー情報
-   利用者設定

キャッシュ禁止:

-   案件情報
-   受付情報
-   SLA状態

理由:

最新状態を表示する必要があるため。

------------------------------------------------------------------------

# 3.10 一覧画面データ取得

一覧画面では必要項目のみ取得する。

取得しない項目:

-   AIResultJson
-   ExceptionJson
-   長文本文

理由:

性能低下防止。

詳細画面のみ取得する。

------------------------------------------------------------------------

# 3.11 詳細画面データ取得

詳細画面表示時に対象レコードを取得する。

取得対象:

-   基本情報
-   AI情報
-   履歴情報
-   関連情報

------------------------------------------------------------------------

# 3.12 AI情報取得方針

AI情報:

-   表示のみ
-   編集不可
-   人による確定値を優先

AIResultJsonは管理者・保守担当のみ参照可能とする。

------------------------------------------------------------------------

# 3.13 排他制御

更新対象データにはUpdatedAt比較を利用する。

処理:

1.  画面表示時UpdatedAt取得
2.  保存時最新UpdatedAt確認
3.  差異ありの場合更新拒否

メッセージ:

「他の利用者によって更新されています。最新情報を取得してください。」

------------------------------------------------------------------------

# 3.14 エラー処理

データ取得失敗:

-   該当画面のみエラー表示
-   再読込可能

更新失敗:

-   入力内容保持
-   エラー通知
-   SystemError記録

------------------------------------------------------------------------

# 3.15 パフォーマンス設計

以下を遵守する。

-   必要列のみ取得
-   不要なRefresh禁止
-   Concurrent活用
-   Delegation警告解消
-   大量Collection作成禁止

------------------------------------------------------------------------

# 3.16 セキュリティ

-   権限確認後に表示
-   非公開データを取得しない
-   AI内部情報を制限
-   Microsoft Lists直接編集禁止

------------------------------------------------------------------------

# 4. 共通UI設計

## 4.1 目的

Power Apps全画面で利用する共通UI仕様を定義する。

-   Fluent UI準拠とする。
-   共通Componentを利用する。
-   画面ごとの独自UI実装を避ける。

  領域         用途
  ------------ ----------------------

  Navigation   メニュー
  Content      業務画面
  Toast        通知

## 4.4 Theme設計

ThemeはRecordとして管理する。

例: - PrimaryColor - BackgroundColor - TextColor - ErrorColor -
WarningColor

## 4.5 Toast仕様

  種類      用途   表示時間
  --------- ------ ----------
  Success   成功   3秒
  Info      情報   3秒
  Warning   警告   5秒

# 07_Power Automate連携設計_Ver1.1

## 1. 目的

本章ではPower AppsとPower Automate間の連携仕様を定義する。

## 2. 基本方針

-   Power Appsは画面制御・入力受付のみ担当する。
-   業務処理、データ更新、AI処理はPower Automateが担当する。
-   フロー呼出はPower AppsのRun()を利用する。
-   CorrelationIdにより処理追跡を行う。

## 3. Run()連携標準

### 呼出形式

``` powerfx
Set(
    varResult,
    FlowName.Run(
        JSON(
            {
                RecordId: ThisItem.RecordId,
                CorrelationId: varCorrelationId
            },
            JSONFormat.Compact
        )
    )
)
```

## 4. 入出力仕様

### Input

  項目            型       説明
  --------------- -------- ----------------
  RecordId        String   対象レコードID
  CorrelationId   String   処理追跡ID
  RequestJson     String   処理依頼内容

### Output

  項目            型        説明
  --------------- --------- ----------------------
  Success         Boolean   処理結果
  Message         String    利用者向けメッセージ
  CorrelationId   String    追跡ID

## 5. 主要連携

  処理       Flow
  ---------- ----------------
  案件化     案件化フロー
  AI再解析   AI再解析フロー

## 6. エラー処理

-   Run()失敗時は画面遷移しない。
-   CorrelationIdを表示する。
-   詳細エラーはSystemErrorで管理する。

## 7. セキュリティ

-   Power Appsから直接Lists更新は禁止。
-   Flow側で権限確認を実施する。
-   AIResultJson等の内部情報は必要利用者のみ参照可能とする。
