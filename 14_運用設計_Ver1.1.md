# 14_00_文書概要_Ver1.0

# 1. 文書概要

## 1.1 目的

本書は、AI運用保守プラットフォームにおける運用設計を定義することを目的とする。

本設計書では、本システムを安定的かつ継続的に運用するための運用方針、運用体制、日次・週次・月次運用、障害対応、AI運用、監視、ログ管理、バックアップ、ユーザー管理、変更管理、定期メンテナンス、運用品質管理およびレビュー方法を定義する。

また、本設計書は運用担当者、運用管理者およびシステム開発者が追加説明なしで運用を実施できる品質を目標とする。

## 1.5 設計方針

-   Microsoft LearnおよびPower Platformベストプラクティスを優先する。
-   約20名規模の社内利用に適したシンプルで保守性の高い運用を採用する。
-   Power Appsは画面制御のみを担当する。
-   Power Automateは業務処理および自動化を担当する。
-   Microsoft ListsをPhase1の唯一の業務データストアとする。
-   AI Builderは業務支援機能として利用し、人による最終判断を前提とする。
-   CorrelationIdおよびFlowRunIdにより業務処理・障害・監査を追跡可能とする。
-   運用作業は標準化し、属人化を防止する。
-   業務停止を最小限とするフェイルセーフ運用を採用する。
-   システム変更は変更管理プロセスを経て実施する。
-   レビュー結果はレビュー管理台帳で管理し、正式版へ直接反映しない。

# 14_01_運用方針_Ver1.0

# 2. 運用方針

## 2.1 目的

本章ではAI運用保守プラットフォームの運用基本方針を定義する。

## 2.2 基本方針

-   Microsoft LearnおよびPower Platformベストプラクティスを採用する。
-   約20名規模を前提としたシンプルな運用とする。
-   Power Appsは画面制御のみ、業務更新はPower Automateが実施する。
-   Microsoft ListsをPhase1の唯一のデータストアとする。
-   AI Builderは補助機能として利用し、人による最終判断を必須とする。
-   CorrelationId・FlowRunIdにより全処理を追跡する。
-   標準化された運用手順を採用し属人化を防止する。

## 2.3 運用目標

  項目           目標
  -------------- --------------------------------
  可用性         業務時間中に安定運用
  保守性         標準手順で対応可能
  追跡性         CorrelationId・FlowRunIdで追跡
  セキュリティ   最小権限運用
  継続性         AI障害時も業務継続

## 2.4 運用原則

1.  業務継続を最優先とする。
2.  AI障害で受付・案件処理を停止しない。
3.  変更は変更管理を経て実施する。
4.  障害は証跡を残し再発防止を行う。
5.  レビュー結果はレビュー管理台帳で管理する。

## 2.5 運用フロー

``` text
監視
 ↓
異常検知
 ↓
一次切り分け
 ↓
復旧対応
 ↓
動作確認
 ↓
原因分析
 ↓
再発防止
```

## 2.6 運用ルール

  項目               内容
  ------------------ --------------
  監視               毎営業日実施
  ログ確認           日次
  権限棚卸           定期実施
  バックアップ確認   定期実施
  レビュー           月次

## 2.7 セキュリティ

-   管理者権限を最小化する。
-   共通アカウントを使用しない。
-   ログを保管し監査可能とする。

## 2.8 将来拡張

Dataverse移行、Copilot連携、Azure AI連携を考慮する。

## 2.9 設計レビュー

  -------------- ------
  要件整合性     適合
  保守性         適合
  セキュリティ   適合

# 14_04_日次運用_Ver1.0

# 5. 日次運用

## 5.1 目的

毎営業日に実施する運用作業を定義する。

## 5.2 日次運用一覧

  作業              担当         タイミング   証跡
  ----------------- ------------ ------------ ----------------
  受付状況確認      運用担当者   始業時       受付一覧
  SLA確認           運用担当者   午前・午後   ダッシュボード
  Flow失敗確認      運用担当者   午前         実行履歴
  SystemError確認   開発者       午前         SystemError
  AI失敗確認        開発者       午前         AI解析状態
  Teams通知確認     運用担当者   随時         Teams

## 5.3 運用フロー

``` text
始業
 ↓
受付確認
 ↓
SLA確認
 ↓
Flow・AI確認
 ↓
障害有無判定
 ├─なし→通常運用
 └─あり→障害運用へ
```

## 5.4 異常時対応

-   Flow失敗は再実行可否を確認する。
-   AI失敗時も受付・案件処理を継続する。
-   必要に応じて運用管理者へエスカレーションする。

## 5.5 チェック項目

-   新規受付遅延なし
-   SLA超過なし
-   Flow失敗未対応なし
-   SystemError未確認なし
-   AI解析滞留なし

# 14_05_週次運用_Ver1.0

# 6. 週次運用

## 6.1 目的

毎週実施する定期運用により、運用品質・データ整合性・システム健全性を維持する。

## 6.2 実施項目

  作業                  担当                   証跡
  --------------------- ---------------------- --------------
  SystemError傾向分析   システム開発者         障害一覧
  フロー失敗件数確認    Power Platform開発者   Flow実行履歴
  SLA超過案件確認       運用管理者             案件一覧
  未対応受付確認        運用担当者             受付一覧
  AI失敗率確認          システム開発者         AI解析ログ

## 6.3 手順

1.  監視結果を確認する。
2.  未対応項目を抽出する。
3.  改善計画を登録する。
4.  運用管理者へ報告する。

## 6.4 判定基準

重大な未対応障害・SLA超過案件が残存しないこと。

## 15.3 ロールバック

## 15.4 Connection Reference・Environment Variables運用

### Connection Reference

-   Power Automate の接続情報は Connection Reference
    を利用して管理する。
-   フローへ個人接続を直接保持しない。
-   接続変更時は影響調査および接続テストを実施する。
-   本番環境へのリリース前に Connection Reference の割り当てを確認する。

### Environment Variables

以下の設定値は Environment Variables で管理する。

-   SharePoint サイト URL
-   Microsoft Lists 名称
-   Teams チャネル ID
-   AI 関連設定値
-   各種環境依存 URL

環境ごとの差異は Environment Variables
により吸収し、フロー定義へハードコードしない。

## 15.5 リリース手順（Power Platform）

1.  Solution のインポート
2.  Environment Variables の設定確認
3.  Connection Reference の再割り当て
4.  接続テスト
5.  Power Automate フロー有効化
6.  Power Apps 動作確認
7.  業務シナリオ確認
8.  本番リリース

## 15.6 ロールバック

重大障害時は以下の手順で復旧する。

1.  対象 Solution をロールバック
2.  Connection Reference を復元
3.  Environment Variables を確認
4.  Power Automate フローを再有効化
5.  動作確認
6.  利用者へ復旧完了を通知
