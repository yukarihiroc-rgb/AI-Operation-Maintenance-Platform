# 1. 文書概要

## 1.1 目的

本書は、AI運用保守プラットフォームにおける障害テスト設計を定義することを目的とする。

Power Apps、Power Automate、AI Builder、Microsoft Lists、SharePoint
Online、Outlook、Microsoft
TeamsおよびSLA管理機能を対象として、障害発生時の業務継続性、フェイルセーフ、障害復旧、再試行処理、データ整合性および監査情報の追跡性を確認するための試験方針、試験環境、試験シナリオ、評価方法および合格基準を定義する。

本設計書は「08_テスト基本設計_Ver1.0」「09_結合テスト仕様書_Ver1.0」「10_受入テスト仕様書_Ver1.0」「11_テストデータ設計_Ver1.0」および「12_性能テスト設計_Ver1.0」を補完する設計書として位置付ける。

## 1.5 設計方針

-   障害発生時でも業務継続可能であることを確認する。
-   AI処理失敗時でも受付登録および案件対応が継続できることを確認する。
-   Retry Policyによる自動再試行が正常に動作することを確認する。
-   フェイルセーフ設計が設計どおり機能することを確認する。
-   CorrelationIdおよびFlowRunIdにより障害発生から復旧まで追跡可能であることを確認する。
-   Microsoft ListsおよびSharePoint Onlineのデータ整合性を確認する。
-   Power Apps・Power Automate・AI Builder・Microsoft
    365各サービス間の障害影響を確認する。
-   Microsoft LearnおよびPower
    Platformベストプラクティスを考慮した試験を実施する。
-   約20名規模の社内利用を想定した現実的な障害試験を実施する。

-   障害試験は再現性を確保する。
-   正常復旧までを試験対象とする。
-   Retry・フェイルセーフ・業務継続性を確認する。
-   データ整合性を維持する。
-   CorrelationIdおよびFlowRunIdにより障害を追跡可能とする。
-   Microsoft標準機能を前提とした障害復旧を確認する。
-   実運用を想定した障害シナリオを採用する。

# 2. 障害テスト方針

## 2.1 目的

本章では、AI運用保守プラットフォームにおける障害テストの基本方針を定義する。

障害発生時においても業務継続性、データ整合性、復旧性および追跡性を維持できることを確認する。

------------------------------------------------------------------------

## 2.2 基本方針

-   実運用を想定した障害シナリオで試験を実施する。
-   Microsoft LearnおよびPower Platformベストプラクティスに準拠する。
-   AI処理失敗時でも受付・案件処理が継続することを確認する。
-   Retry Policyによる自動再試行を確認する。
-   フェイルセーフ設計どおりの動作を確認する。
-   CorrelationIdおよびFlowRunIdにより障害を追跡可能であることを確認する。
-   障害復旧後にデータ整合性が維持されることを確認する。

------------------------------------------------------------------------

## 2.3 試験対象

-   Power Apps
-   Power Automate
-   AI Builder
-   Microsoft Lists
-   SharePoint Online
-   Outlook
-   Microsoft Teams
-   SLA管理
-   SystemError
-   Retry処理
-   CorrelationId・FlowRunId

------------------------------------------------------------------------

## 2.4 試験観点

  観点       確認内容
  ---------- ------------------------------
  業務継続   障害時も受付・案件処理継続
  復旧性     障害復旧後に正常動作
  再試行     Retry Policyによる自動復旧
  整合性     データ欠損・重複なし
  通知       障害通知・SLA通知
  AI         AI障害時のフェイルセーフ
  追跡       CorrelationId・FlowRunId追跡
  監査       SystemError記録

------------------------------------------------------------------------

## 2.5 合格基準

-   業務停止が発生しないこと
-   自動復旧対象はRetryで復旧すること
-   復旧後のデータ整合性が維持されること
-   SystemErrorへ障害情報が記録されること
-   CorrelationIdおよびFlowRunIdで追跡可能であること
-   利用者へ適切なエラーメッセージが表示されること
