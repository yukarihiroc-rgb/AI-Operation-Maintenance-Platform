## 2. 目的

本書は、AI運用保守マネジメントプラットフォーム Phase1におけるシステム構成、メール受付、AI判定、データ管理、案件管理、通知、SLA、障害対応、バックアップ・復旧、権限、運用保守、テストおよびリリースの基本方針を定義する。

具体的なMicrosoft Listsのカラム定義、AI Builderの解析項目・JSONスキーマ、Power Appsの詳細画面レイアウト、Power Automateの詳細フロー、リトライ回数、バックアップ頻度等は、後続の詳細設計で定義する。

---

## 3. 基本方針

Phase1では、Microsoft 365およびPower Platformを中心として、Outlookで受信したメールをPower Automateで取得し、メールメタデータおよびAI Builderによる解析結果をMicrosoft Listsへ登録する。

SEはPower Appsを統合GUIとして利用し、受付情報、案件情報、SLA、通知、エラー情報等を確認・更新する。

初期リリースを過度に遅らせず、必要な機能を実装して限定利用から開始し、運用開始後のSEからのフィードバックを反映して継続的に改善する。

Phase1では外部システムとのAPI連携等は実装対象としない。将来拡張については、実運用開始後の改善要求および運用実績を基に検討し、現時点では具体的な将来拡張機能を基本設計の対象としない。

## 3.1 用語定義・現行運用との対応

| 用語 | 定義 |
|---|---|
| 受付 | Outlookで受信したメールを起点として本プラットフォームに登録された情報。原則1メールにつき1受付とする。 |
| 案件 | 本プラットフォームにおいて継続的な管理対象として登録された課題・依頼・障害等の総称。現行運用の課題管理簿・依頼管理簿・障害管理簿等で管理される個別管理対象に相当する。 |
| 案件化 | 受付情報を人が確認し、継続的な管理が必要と判断して新規案件として登録すること。現行運用における各管理簿への記載に相当する。 |
| 案件化判断 | 受付について、新規案件化、既存案件への関連付け、案件化不要のいずれとするかを人が判断すること。複数対象システムがある場合は対象システム単位で行う。 |
| 案件化判断完了 | 必要な案件化判断が完了した状態。新規案件化と同義ではない。 |
| 未案件化 | Power Apps上の利用者向け表示用語。内部状態は「未判断／判断済み」で管理する。 |
| 案件化不要 | 人が確認した結果、新規案件化および既存案件への関連付けが不要と判断した状態。後日案件化可能とする。 |
| 対応履歴 | 案件に対して実施した調査、回答、作業等の対応実績を時系列で記録した情報。 |
| 変更履歴 | 案件属性、対応内容、対応履歴の無効化・再有効化等、案件に関連する重要な変更を記録した情報。 |

| メール受信・問い合わせ受領 | 受付 |

## 4. システム全体構成

### 4.1 基本構成

```text
Outlook
  ↓
Power Automate
  ↓
メールメタデータ取得
  ↓
AI Builderによる内容解析
  ↓
JSON解析
  ↓
Microsoft Lists（Phase1物理13リスト）
  ├─ 受付情報リスト（IntakeInfo）
  ├─ 受付対象システムリスト（IntakeTargetSystem）
  ├─ 受付案件関連リスト（IntakeCaseRelation）
  ├─ 案件情報リスト（CaseInfo）
  ├─ 案件担当者リスト（CaseAssignee）
  ├─ 案件対応履歴リスト（CaseActionHistory）
  ├─ 案件変更履歴リスト（CaseChangeHistory）
  ├─ AI類似提案履歴リスト（AISimilarityHistory）
  ├─ AIプロンプト変更履歴リスト（AIPromptChangeHistory）
  ├─ システムエラーリスト（SystemError）
  ├─ 共通マスタリスト（CommonMaster）
  ├─ システム設定リスト（SystemConfig）
  └─ 担当者関連リスト（PersonAssignment）
  ↓
Power Apps
  ├─ 運用用ダッシュボード
  ├─ 受付一覧
  ├─ 案件一覧
  ├─ クローズ関連
  ├─ SLA管理
  ├─ システム保守用ダッシュボード
  ├─ システムエラー
  └─ 各種設定

通知先
  ├─ Outlookメール
  └─ Teams
```

詳細な論理データ構成および物理Microsoft Lists構成は、「02_データ設計」を参照する。

### 4.2 AI実装方式

Phase1のAI解析にはAI Builderを使用する。

基本的な処理順序は以下とする。

```text
メール受信
  ↓
メールメタデータによる機械判定
  ↓
AI Builderによる内容判定
  ↓
判定結果をJSON形式で処理
  ↓
Microsoft Listsへ反映
  ↓
判定不能の場合は人へTeams通知
```

具体的なAI解析項目、プロンプト、判定条件、JSONスキーマ等は詳細設計で定義する。

## 5. メール受付方式

### 5.1 受付メールボックス

PoCでは個人の受信ボックスをPower Automateのトリガーとして使用したが、本番運用では個人メールボックスへの依存を避ける。

共有メールボックスの利用可否についてMicrosoft 365のシステム担当者へ確認中である。共有メールボックスの利用が可能な場合は共有メールボックスを利用し、困難な場合はシステム管理者のメールボックスで管理する方針とする。

既存のメーリングリストは以下を想定する。

- 新システム：新システムのメーリングリスト
- 保システム：保システムのメーリングリスト
- Iシステム：Iシステムのメーリングリスト
- Cシステム：Cシステムのメーリングリスト
- 保守向け共有：保守全体のメーリングリスト

### 5.2 1メール＝1行

受信したメールは、原則として1メールにつきMicrosoft Listsの受付情報1行として登録する。

ただし、1メール＝1受付案件とはしない。リマインドメール、SEからの返信メール、その他の関連メールも履歴として登録し、関連する起点メールへ紐付ける。

### 5.3 受付No

メール受信時点で、すべてのメールに対して受付Noを発番する。

受信後に既存受付との関連が判明した場合も、発番済み受付Noは統合・無効化せず保持する。

### 5.4 メール関連付け・スレッド管理

関連メールは、最初の起点メールへ直接紐付ける1階層構造とする。

```text
起点メール：受付No 1001
  ├─ 関連メール：受付No 1002 → 親受付No 1001
  ├─ 関連メール：受付No 1003 → 親受付No 1001
  └─ 関連メール：受付No 1004 → 親受付No 1001
```

親→子→孫の多階層構造は採用しない。

起点メール自身の親受付Noは空欄とする。

Power Appsでは、Outlookのスレッド表示に近い形で、一連の関連メールを確認可能とする。

関連判定にはOutlookのメール識別情報、メッセージID、会話関連のメタデータ等の利用を考慮する。具体的なメタデータ、判定キー、関連付けロジックは詳細設計で定義する。

## 6. メール判定・通知方針

### 6.1 判定順序

メール判定は以下の順序とする。

1. メールメタデータによる機械判定
2. AI Builderによる内容判定
3. 判定不能の場合はシステム担当者へTeams通知し、人による確認を依頼

### 6.2 基本分類

少なくとも以下の分類を想定する。

- 新規受信
- リマインドメール
- その他の関連メール
- 判定不能

具体的な判定条件は詳細設計で定義する。

### 6.3 通知方針

| 判定・状態 | 通知方針 |
|---|---|
| 新規受信として正常受付 | 担当システムのメーリングリストへ新規受信の受付完了メールを送信 |
| リマインドメール | 担当システムへTeams通知 |
| 至急対応を要するメール | 担当システムへTeams通知 |
| 判定不能 | システム担当者へTeams通知 |
| 担当システムを特定できない | 全体チャットへ通知 |

ユーザー向けの自動返信機能はPhase1では実装しない。実装する場合はPhase2以降とする。

## 7. Microsoft Lists構成

### 7.1 リスト分割方針

受付情報と案件情報は別々のMicrosoft Listsで管理する。

- 受付情報リスト
- 案件情報リスト
- 受付案件関連リスト

受付情報はメール単位の履歴管理を目的とし、案件情報はSEによる対応・進捗・担当者・ステータス等の案件管理を目的とする。

受付と案件の関係は多対多とし、1つの受付情報から複数の案件を作成可能とする。また、複数の受付情報を同一案件へ関連付け可能とする。

受付情報と案件情報の関連は、「受付案件関連リスト」で管理する。

具体的なカラム定義はデータ詳細設計で定義する。

### 7.1.1 データ名称・命名規則

- Microsoft Lists上のデータ格納単位は「リスト」と呼称する。
- リスト内の1件のデータは「レコード」、各データ項目は「列」と呼称する。
- 設計書および利用者向けには日本語の論理名を使用する。
- Power Apps、Power Automate、Microsoft Graph、JSON等から参照する物理名は、英数字のPascalCaseを使用する。
- Microsoft Listsを指して「DB」と呼称しない。
- 旧名称は新旧対照表でのみ参照し、本文は新名称へ統一する。

### 7.1.2 物理Microsoft Lists構成

本プラットフォームでは、論理データ領域と物理リストを区別し、Phase1の物理Microsoft Listsを以下の13リストで構成する。

| No. | 論理名 | 物理名 |
|---:|---|---|
| 1 | 受付情報リスト | `IntakeInfo` |
| 2 | 受付対象システムリスト | `IntakeTargetSystem` |
| 3 | 受付案件関連リスト | `IntakeCaseRelation` |
| 4 | 案件情報リスト | `CaseInfo` |
| 5 | 案件担当者リスト | `CaseAssignee` |
| 6 | 案件対応履歴リスト | `CaseActionHistory` |
| 7 | 案件変更履歴リスト | `CaseChangeHistory` |
| 8 | AI類似提案履歴リスト | `AISimilarityHistory` |
| 9 | AIプロンプト変更履歴リスト | `AIPromptChangeHistory` |
| 10 | システムエラーリスト | `SystemError` |
| 11 | 共通マスタリスト | `CommonMaster` |
| 12 | システム設定リスト | `SystemConfig` |
| 13 | 担当者関連リスト | `PersonAssignment` |

各種マスタは論理分類を維持しつつ、物理的には共通マスタリストへ統合する。担当者マスタも論理データ領域として扱い、担当者本人の基本情報は共通マスタリスト（`CommonMaster`）内で担当者区分により識別して保持する。担当者と対象システム・担当チーム・役割の組み合わせ関係は、担当者関連リスト（`PersonAssignment`）で別管理する。システム動作を制御する設定値は、共通マスタとは分離してシステム設定リストで管理する。

### 7.2 設定情報の外出し

以下のような運用上変更される情報は、Power Automateのフローへ直接埋め込まず、Microsoft Lists等の設定情報として外出しし、Power Appsから変更可能とする。

- SLA設定
- 通知先
- 営業日・休日
- 権限・役割情報
- その他の運用設定

これにより、設定変更時にPower Automateのフロー修正・再リリースを不要とする。

### 7.3 対応履歴

対応履歴はPhase1から追記型で蓄積し、案件と1対多関係とする。

1つの対応履歴には、主対応者1名および共同対応者複数名を設定可能とする。

SEは対応内容を記録し、AI Builderはその対応内容を解析して、対応種別マスタおよび対応結果マスタから、それぞれ代表的な値を1つ判定する。

SEはPower Apps上で対応履歴の「対応内容」を編集可能とする。

対応内容が変更された場合は、変更前後の内容、変更日時、変更者等を「案件変更履歴リスト」で一元管理し、自動的に追記保存する。

案件変更履歴リストに記録される対応内容変更の履歴はPower Appsには表示しないシステム内部ログとし、通常のSEは編集・削除できない。保守上必要な場合に限り、システム管理者が修正可能とする。

### 7.4 対応種別マスタ

対応履歴における「何を行ったか」を分類するため、対応種別をマスタ化する。

初期値として以下を想定する。

- 調査
- 作業
- ユーザー確認
- ユーザー回答
- 内部確認
- 対応方針決定
- 復旧
- 経過観察
- 問い合わせ
- エスカレーション
- 訂正
- その他

対応種別マスタには、対応種別コード、対応種別名、定義、AI判定基準、有効・無効、表示順等を保持する。

### 7.5 対応結果マスタ

対応履歴における「その対応によってどのような結果となったか」を分類するため、対応結果をマスタ化する。

初期値として以下を想定する。

- 解決
- 復旧
- 原因不明・復旧
- 暫定対応
- 一部解決
- 継続対応
- 原因特定
- 原因不明
- 情報取得
- 回答済み
- 対応対象外
- 起案化
- 判定不能

「再現せず」および「事象確認できず」は「原因不明」へ集約する。

「問い合わせ」および「エスカレーション」は対応行為であるため、対応結果マスタには含めず、対応種別マスタで管理する。

対応結果マスタには、対応結果コード、対応結果名、定義、AI判定基準、有効・無効、表示順等を保持する。

1つの対応履歴につき、対応種別1つ、対応結果1つを保持する。

### 7.6 対応履歴のファイル・ドキュメント管理

エラー画面等のスクリーンショットは、その時点の事象を示す証跡として対応履歴へ直接添付可能とする。

以下のファイルはSharePointドキュメントライブラリへ保存し、対応履歴からリンクで参照する。

- ログファイル
- 一時的な調査資料
- 正式な手順書
- 設計書
- 大容量ファイル
- その他、原本一元管理が適切なドキュメント

1つの対応履歴には、複数のSharePointリンクを保持可能とする。

同一ドキュメントを対応履歴ごとに複製せず、SharePoint上の原本を参照することで、重複データの発生を防止し、手順書等の改訂時に個別再展開を不要とする。

### 7.7 ドキュメント種別マスタ・AI自動分類

SharePointの指定ドキュメントライブラリへファイルがアップロードされた場合、Power Automateが登録を検知し、AI Builderによる解析を実行する。

AI Builderは、ファイル名および取得可能な文書内容等を基に、ドキュメント種別マスタから代表的な種別を1つ判定し、SharePointのメタデータへ自動設定する。

初期のドキュメント種別として以下を想定する。

- 手順書
- 設計書
- ログ
- 調査資料
- テスト資料
- 変更関連資料
- ナレッジ
- その他
- 判定不能

AIが新しい種別を自由生成する方式とはせず、ドキュメント種別マスタに登録された候補から選択する。

AIによる誤判定があった場合は、人が修正可能とする。

ドキュメント種別マスタ自体の追加・変更・無効化は、権限を持つ管理者が行う。

具体的な対象ファイル形式、本文抽出方式、AIプロンプト、エラー時の処理、再解析方式等は後続の詳細設計で定義する。


### 7.8 AI生成・判定項目に対する人の修正優先

AIが生成または判定した情報を人が修正・確定した場合、人による値を正として扱う。

人が修正・確定した値は、AIによる自動再解析、再実行または後続処理によって自動上書きしない。

AI初回判定値と人による修正後の確定値は、AI判定品質の確認および改善に利用できるよう保持する。

AIは過去の人による修正内容を参照情報として活用し、次回以降の判定・提案品質の改善に利用する。ただし、AI Builder等のAIサービス自体が自動学習することを前提とはしない。

具体的な参照方法、プロンプトへの反映方法、参照対象範囲、保持項目等は後続の詳細設計で定義する。

本方針は、案件件名、対象システム、案件種別、優先度、対応種別、対応結果、ドキュメント種別その他のAI生成・AI判定項目に適用する。

### 7.9 担当者マスタ

運用保守メンバーを論理上の担当者マスタで管理する。担当者本人の基本情報は、物理的には共通マスタリスト（`CommonMaster`）内で担当者区分により識別して保持する。

登録対象はMicrosoft 365ユーザーのみとする。

担当者の初回登録時は、Power Apps等からMicrosoft 365組織内ユーザーを検索・選択し、表示名、メールアドレスその他の取得可能な基本情報を自動取得する方式を第一候補とする。

本機能は現行Microsoft 365環境での実機検証を必要とする。自動参照機能を実装できない場合は、担当者マスタを人が手動で作成・管理する代替方式とする。

初回登録後は、Microsoft 365ユーザー情報との定期同期は行わない。

担当者は削除せず、有効・無効で管理する。無効化後も過去の案件および対応履歴との関連を保持する。

### 7.10 担当者の担当範囲・役割

1人の担当者は、以下を複数保持可能とする。

- 担当チーム
- 対象システム
- 役割

主担当システムは0件以上設定可能とする。

担当者と対象システムの関連情報には、少なくとも対象システムおよび主担当フラグを保持する。

役割は担当チームとの組み合わせで管理可能な構造とし、同一人物がチームごとに異なる役割を持てるようにする。

具体的なリスト分割、関連情報の保持方式およびカラム定義はデータ詳細設計で定義する。

### 7.11 担当者の参画日・退場日と自動有効化・無効化

担当者マスタには参画日および退場日を保持する。

担当者の有効状態は以下のルールで自動制御する。

- 参画日前：無効
- 参画日当日：自動的に有効化
- 参画中：有効
- 退場日当日：有効
- 退場日の翌日以降：自動的に無効化
- 退場日未設定：参画日以降は有効を維持

自動制御はPower Automate等による定期処理を第一候補とし、具体的な実行頻度およびフロー構成は詳細設計で定義する。

担当者の所属履歴を別データとして厳密に管理する方式は採用しない。

再参画時は既存の担当者マスタを再利用し、参画日、退場日および有効状態を更新する。過去の対応実績は対応履歴側で保持する。

## 10. Teams通知

### 10.1 通知単位

通知は基本的に担当チーム単位とする。

例として、新システムに関する通知は、新システムのリーダーおよびメンバーへ通知する。

担当システムを特定できない場合は全体チャットへ通知する。

### 10.2 通知先設定

通知対象はTeamsアカウント単位で個別設定可能とする。

例えば、チーム通知の場合はAさん、Bさん、CさんのTeams IDを設定し、リーダーのみへ通知する場合はAさんのみを設定できる構成とする。

通知先はPower Automateのフローへ直接埋め込まず、設定情報として外出しし、Power Appsから変更可能とする。

### 10.3 Power Appsへの直接遷移

Teams通知から、該当する受付または案件をPower Appsで直接開ける構成とする。

## 11. 案件管理

### 11.1 案件化

受付時に種別の振り分けを行い、誤りがある場合はSEが修正可能とする。

受付に対する案件化判断は対象システム単位で行う。最終判断は、原則として当該対象システムを担当するチームが行う。基本的にはリーダーによる判断を想定するが、休暇等を考慮し、担当チームへ案件化確認通知を行う。

案件化時に、課題・依頼等の分類に応じて課題No、依頼No等を割り当てる。

受付時の分類と案件化後の分類は分離する。

### 11.2 担当者アサイン

案件化時に人が担当者をアサインする。

担当チームへ通知後、リーダーまたはメンバーが担当者を設定する。

複数担当者を設定可能とするが、主導者不在を防止するため、運用上は基本1名の担当者を推奨する。

### 11.3 完了・差し戻し・クローズ

メンバーを含む利用者が案件を「完了」にできる。

完了後は担当システムのリーダーによる確認を想定する。情報不足等の場合は「差し戻し」とし、差し戻し理由を記録する。

クローズ権限自体は特定役割へ限定せず、利用者が実施可能とする。

システム管理者が未確認のクローズ案件を識別できるよう、別タブ等で表示可能とする。顧客とシステム管理者間でのクローズチェック完了状況を識別可能なステータス等を設ける。

### 11.4 再オープン

クローズ済み案件は全員が再オープン可能とする。

再オープン時はステータスを「再オープン」とし、「対応中」には戻さない。

再オープン理由を記載可能とする。

差し戻し理由と再オープン理由については、Power Apps上で同一の入力UIを流用することを考慮する。

### 11.5 クローズ案件の表示

クローズ済み案件は通常の案件一覧から除外するが、データ自体は削除しない。

再オープンした場合は通常の案件一覧へ再表示する。

---

## 11.6 受付の案件化判断と表示

人による振り分け完了前は未案件化状態とし、AI提案値は参考値であることを明示してPower Appsに表示する。人が確定・修正した後は人の現在確定値を通常画面に表示し、AI初回判定値は内部データとして保持する。

受付に複数対象システムがある場合は対象システム単位で案件化判断を行う。対象システム0件も許可し、人が案件化不要と判断可能とする。一度案件化不要と判断した受付も後日案件化可能とする。

判断済み受付への後日の対象システム追加では、受付全体を自動的に未判断へ戻さない。新規システムは原則としてプラットフォームへの定義後に受付対象とする。

## 12. 変更履歴

案件属性変更と対応内容変更は、個別の履歴リストを乱立させず、共通の「変更履歴」で一元管理する。少なくとも案件ステータス、案件対象システム、案件種別、優先度、主担当者、副担当者、主担当チーム、関連チーム、対応内容、対応履歴の無効化・再有効化等を履歴対象とする。

変更履歴では、対象データ、変更種別、変更前の値、変更後の値、変更日時、変更者等を追跡可能とする。ただし、受付対象システムリストのように関連レコード自体を削除せず無効化・新規追加することで変更経緯を追跡できる場合は、同一情報を変更履歴へ重複保存しない。

現時点では変更履歴の保存期間を設定せず保持し、将来データ量や利用状況に応じて削除条件・保持期間を設定可能な構成とする。それ以外の変更についてはMicrosoft Lists標準のバージョン履歴を利用する。

本システムは内部向け運用保守アプリケーションであり、厳格な監査対応を目的とはしないが、問題発生時に変更内容を確認できるよう適切な変更履歴管理を実装する。

## 14. バックアップ・復旧

### 14.1 重要データ

以下を重要なバックアップ対象とする。

- 受付情報
- メール履歴
- 案件情報
- 重要操作の変更履歴
- SLA設定
- 営業日カレンダー
- 各種マスタ
- 権限・通知先等の設定情報

システムエラー情報はPhase1初期の障害分析・改善に利用するため一定期間保持する。

### 14.2 基本バックアップ方針

Microsoft Listsの標準的なバージョン履歴および削除データの復旧機能を基本とし、重要業務データについては定期的な自動バックアップを実施する。

受付情報および案件情報は最重要業務データとして扱い、データ損失を可能な限り最小化する。

当日分を含め、受付情報の消失を極力防止する。

具体的なバックアップ頻度、保持世代、保存先および復旧手順は詳細設計で定義する。

### 14.3 障害復旧能力要件

**障害発生前の正常なバックアップ時点から、Outlook上に保持されている元メールを基に受付処理を再実行可能とする。**

システムエラー情報から障害発生時刻およびエラー発生処理を特定し、復旧対象期間を判断できる構成とする。

復旧時は、障害発生前の正常なバックアップ時点を基準としてデータを復旧し、その時点以降のOutlookメールを対象として受付処理を再実行する。

```text
障害発生
  ↓
システムエラー情報から障害発生時刻・処理を確認
  ↓
障害発生前の正常なバックアップ時点を特定
  ↓
バックアップデータを基準に復旧
  ↓
Outlookから対象期間のメールを再取得
  ↓
受付処理を再実行
  ↓
受付情報を再構築
```

再実行時は、メール識別情報等を利用し、Microsoft Listsへの二重登録、重複更新および重複通知を防止する。

具体的なバックアップ頻度、復旧基準時刻の決定方法、メール再取得範囲、再処理方式、重複判定方法およびリラン処理は機能詳細設計で定義する。

### 14.4 Outlookメールの位置付け

Outlook上の元メールは保持されていることを前提とし、受付情報復旧時の再処理元として利用する。

案件情報は、人による担当者設定、ステータス変更、差し戻し理由、再オープン理由等、元メールだけでは完全復旧できない情報を含むため、確実なバックアップ対象とする。

## 15. 可用性・業務継続

システム停止中もOutlookによるメール受信は継続する。

Power AppsまたはPower Automate等に障害が発生した場合、SEは従来どおりOutlookでメールを確認し、直接返信して業務を継続する。

障害中に発生した案件情報は、原則としてシステム復旧後にPower Appsへ更新する。

長時間障害となる場合は、必要に応じてExcel等へ暫定的に記録し、復旧後に各SEがPower Appsへ反映する。これは暫定的な障害対応とし、詳細な手順は実際の障害発生時に必要に応じて別途検討する。

システム障害中もSLAは継続する。

復旧後は、障害発生前の正常なバックアップ時点からOutlookメールの受付処理を再実行し、受付情報を再構築する。

障害中にSEがOutlookから送信した返信メールについても、関連メール履歴として再構築可能な構成を考慮する。

## 18. セキュリティ・アクセス制御

### 18.1 Power Apps利用者

Power Appsは、明示的に利用権限を付与された以下の運用保守メンバーのみ利用可能とする。

- SE
- リーダー
- SM
- PM

未許可ユーザーが勝手にシステムへ接続できないようにする。

### 18.2 業務データの閲覧・編集

一般SEを含む利用者は、全システムの受付・案件を閲覧・編集可能とする。

システム兼任や少人数体制を考慮し、システム個別の厳密な権限制御は行わない。

### 18.3 Power Apps内の簡易権限制御

本システムは限定された運用保守メンバーが利用する内部システムであるため、Power Apps内の機能権限制御は、ユーザーの役割に応じたボタン、タブ、画面等の表示・非表示を基本とする。

一般SEには管理機能を非表示とし、必要な役割を持つユーザーへ表示する。

### 18.4 Microsoft Lists直接アクセス

Microsoft Listsへの直接アクセスは原則として許可しない。

直接アクセスは以下に限定する。

- 開発者
- 個別に権限を付与されたユーザー

知らないユーザーがDB相当のMicrosoft Listsを直接編集できない構成とする。

### 18.5 設定変更権限

SLA、マスタ、通知先等の日常的な設定変更は、リーダー、SM、PM以上が実施可能とする。

権限は担当システム単位に限定せず、システム横断とする。

### 18.6 履歴閲覧

- システム全体の変更履歴：開発者が閲覧可能
- 専用のマスタ変更履歴は設けない。

## 19. 運用・保守体制

| 項目 | 担当・方針 |
|---|---|

ただし、将来的に業務範囲、利用者数、データ量またはシステム重要度が拡大した場合は、変更管理および承認プロセスの導入を含め、運用・保守体制を適宜見直す。

## 20. 開発・テスト・リリース環境

### 20.1 環境構成

専用の開発環境・テスト環境は設けず、実際に業務利用されているMicrosoft 365環境内で開発、テストおよび本番運用を行う。

顧客から専用テスト環境、テスト用Microsoft 365アカウントおよびライセンスは提供されていない。

開発中は、Power Apps、Power Automate、Microsoft Lists等を可能な範囲で論理的に区別し、実業務データへの意図しない影響を防止する。

### 20.2 テストデータ

過去の実メールをサンプルとして使用する。

必要に応じてテスト用に加工したデータを使用する。

### 20.3 テスト方針

| テスト区分 | 方針 |
|---|---|
| 機能テスト | 実施する。各機能が要件どおり動作することを確認する。エラー検出、自動リトライ、手動リラン、重複防止、バックアップからの受付再構築等、実装した復旧機能も機能テスト対象とする |
| サービス間結合テスト | 実施する。Outlook、Power Automate、AI Builder、Microsoft Lists、Power Apps、Teams等のサービス間連携を確認する |
| 業務シナリオテスト | 実施する。メール受信から受付、案件化、対応、完了、クローズ等の一連の業務フローを確認する |
| 障害試験 | 実施しない |
| 性能テスト | 実施しない。ただし、通常の機能テスト・結合テストの中で明らかな性能問題がないことは確認する |
| 運用テスト | 独立したテスト工程としては実施しない |

障害試験は実施しないが、障害発生時の復旧を目的として実装した機能については、通常の機能テストとして動作確認を行う。

## 25. まとめ

Phase1では、Outlook、Power Automate、AI Builder、Microsoft Lists、Power Apps、Teamsを中心として、メール受付からAI解析、受付・案件管理、通知、SLA管理、エラー処理、復旧までを一体化した内部向け運用保守プラットフォームを構築する。

特に、以下を重要な基本方針とする。

- 1メール＝1行で受付情報を保持する
- すべてのメールに受付Noを発番する
- 関連メールは起点メールへ直接紐付ける1階層構造とする
- メタデータ判定、AI Builder判定、人による確認の順で処理する
- 受付情報と案件情報は別々のMicrosoft Listsで管理し、受付案件関連リストを介して多対多で関連付ける。1つの受付から複数案件を作成可能とし、複数受付を同一案件へ関連付け可能とする
- メール本文全文はMicrosoft Listsへ保存せず、AI要約とOutlookメール識別情報を保持する
- 対応内容はPhase1から追記型の対応履歴として蓄積する
- SEはPower Apps上で対応履歴の「対応内容」を編集可能とする
- 対応内容の変更時は、変更前後の内容、変更日時、変更者等を「案件変更履歴リスト」へ自動記録する
- 案件変更履歴リストに記録される対応内容変更の履歴はPower Appsには表示しないシステム内部ログとし、通常のSEは編集・削除不可、システム管理者のみ保守上必要な場合に修正可能とする
- 対応履歴は主対応者1名・共同対応者複数名を保持し、AIが対応種別と対応結果をそれぞれ1つ判定する
- エラー画面等のスクリーンショットは対応履歴へ直接添付し、ログ・調査資料・手順書・設計書等はSharePoint原本への複数リンクで参照する
- SharePoint指定ライブラリへのファイル登録時に、AIがドキュメント種別を自動判定してメタデータへ付与する
- 最終SE更新日時は対応内容の追記時のみ更新し、自動処理や管理情報変更と区別する
- 案件種別、対象システム、担当チーム、案件ステータス、優先度、受付種別、対応種別、対応結果、ドキュメント種別、担当者をマスタ化する
- AI生成・判定値を人が修正・確定した場合は人の値を正とし、AIは自動上書きしない
- 人による修正内容は、次回以降のAI判定・提案品質改善のための参照情報として活用する
- 担当者はMicrosoft 365ユーザーのみを対象とし、初回登録時のユーザー自動参照を第一候補とする。実装できない場合は手動マスタ管理へ切り替える
- 担当者は複数チーム・複数システム・複数役割を保持可能とし、主担当システムも0件以上設定可能とする
- 担当者は参画日・退場日に基づいて自動的に有効化・無効化し、無効化後も過去の案件・対応履歴との関連を保持する
- 優先度と緊急度は優先度へ統合する
- Power Appsは1つの統合アプリとする
- 運用用ダッシュボードとシステム保守用ダッシュボードを分離する
- SLA、営業日、通知先等の変更可能な設定は外出しする
- システムエラーはPower Appsから確認し、手動リラン可能とする
- 自動リトライ・手動リラン時はデータ不整合を防止する
- 障害発生前の正常なバックアップ時点から、Outlook上の元メールを基に受付処理を再実行可能とする
- システム停止中もOutlookによる手動運用を継続し、SLAは停止しない
- 業務データは原則として期限を設けず長期保持する
- Microsoft Listsへの直接アクセスは開発者および個別許可ユーザーに限定する
- Phase1は限定ユーザーから利用を開始し、Outlookとの目視突合を行う
- 過去データは初期移行せず、リリース後に本システムのデータ構造へ合わせて取り込む

以上。
